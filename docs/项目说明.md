# 新媒体信源采集 · 项目说明

**撰写人：** 周书伟
**日期：** 2026-06-10

---

## 一、项目是什么

这是一套**本地运行的自动化工具**，用来把四个新媒体平台——**B 站、微博、抖音、快手**——上指定博主主页的内容数据，自动采集下来并整理成 Excel 表格。

一句话概括：**把原来需要人手在采集软件里一条条点击、导出、改名的活，变成机器每天自动完成。**

---

## 二、为什么要做

### 现状的痛点

目前四个平台的"博主主页内容"数据，是靠人在某个图形采集工具上手动操作：逐个平台打开、点击采集、导出成 Excel，再按当天日期手工命名归档（例如 `B站UP主主页视频采集 - 5.13.xlsx`、`抖音-博主主页视频采集 - 5.20.xlsx`）。

这种方式有几个明显问题：

1. **纯靠人工**——每天都要有人重复操作，占用人力。
2. **容易出错**——漏采、命名不统一、平台之间靠人脑记忆同步，难免遗漏。
3. **换台电脑就不灵**——操作流程和工具配置都在某个人的电脑上，别人接手要从头摸索，不可复现。
4. **不可追溯**——一旦中间断了几天，没人说得清哪些数据采过、哪些没采。

### 这个工具的价值

- **省人力**：从"每天手动点"变成"每天自动跑"，人只需要每周看一眼运行状态。
- **更可靠**：用程序按固定逻辑采集，不会漏、不会重，命名风格统一。
- **可移植**：换任何一台电脑，下载代码、装好环境、扫码登录一次，就能立刻接着用。
- **可交接**：整个流程沉淀成代码和文档，不再依赖某个具体的人。

## 三、核心约束（项目的设计底线）

| 约束 | 说明 |
|---|---|
| **完全免费** | 不购买付费代理、不接商业反爬服务、不上付费云服务。只用平台公开接口。 |
| **尽量少人工** | 目标是每天自动跑、人每周看一次。除了登录态过期需要重新扫码，没有日常操作。 |
| **任何电脑可用** | 工具本身可移植，下载 + 安装 + 每平台扫码一次即可运行，Windows / Mac 都支持。 |
| **登录信息不外传** | 各平台的登录凭据（cookie）只存在本机，不上传、不进代码库，杜绝账号安全风险。 |
| **不接外部通知** | 不发邮件、不接推送。运行是否正常，通过一个本地"健康面板"命令查看。 |

## 四、整体方案

### 工作流程

```
   配置要采的博主         自动采集              整理落盘             生成报表
 ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
 │ 列出每个平台 │    │ 程序逐个访问 │    │ 原始数据存成 │    │ 渲染成 Excel │
 │ 要采的账号   │──▶│ 平台接口抓取 │──▶│ JSON 文件    │──▶│ 命名风格保持 │
 │ （一份清单） │    │ 只抓新内容   │    │ （可追溯）   │    │ 与现有一致   │
 └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### 四个关键设计

**1. 直接对接平台接口，而不是模拟点击**

不再依赖第三方采集软件去"模拟人点击网页"，而是直接请求平台官方的数据接口。这样更快、更稳、数据更完整。代价是每个平台的接口和它的"签名验证"机制各不相同，需要分别对接（这也是工作量的主要来源）。

**2. "只采新的"——增量采集**

每条采过的内容都会存成一个独立文件。下次再跑时，程序从最新的内容往回翻，一旦遇到"这条已经存过了"，就停下来。这样每天只抓增量，又快又不会重复，而且即使中途断了，下次接着跑也不会乱——**文件本身就是进度记录，无需额外维护状态**。

需要时也可以强制"全量重采"。

**3. 原始数据先存档，再生成报表**

采到的数据先原封不动存成 JSON 存档，Excel 是从存档**二次渲染**出来的。好处是：万一以后报表格式要调整，不用重新采，直接拿存档重新生成即可。Excel 的列和命名风格会**刻意对齐现有的手工归档习惯**，方便和历史数据无缝衔接。

**4. 自动定时运行**

工具会配上"开机定时任务"（Mac、Windows、Linux 各有对应方案），让它每天在固定时间自己跑起来，无需有人守着。

## 五、四个平台怎么处理

四个平台共用同一套采集"骨架"（登录 → 抓取 → 解析 → 存档 → 渲染），但每个平台接口细节、反爬验证、数据字段都不一样，需要单独适配：

| 平台 | 主要难点 |
|---|---|
| **B 站** | 接口带 `wbi` 签名验证，需还原其算法 |
| **抖音** | 接口带 `X-Bogus` 签名验证，是四个里最复杂的 |
| **微博** | 相对直接，主要是数据字段解析 |
| **快手** | 接口形式特殊（GraphQL），需对应处理 |

登录统一用"弹出浏览器扫码"的方式：运行登录命令 → 弹出干净的浏览器 → 手机扫码 → 程序自动捕获登录态并存到本机。每个平台扫一次，之后就能自动跑，直到登录过期再重扫。

## 六、可靠性与安全

- **运行健康面板**：一条命令即可查看四个平台最近一次跑的情况——采了多少条、有没有报错、登录态是否快过期。这是日常唯一需要人看的东西。
- **出错不中断**：某个平台或某个账号采集失败，不会影响其他的，错误会记录下来供排查。
- **凭据安全**：登录信息只存本机、明确排除在代码库之外，不存在泄露到 GitHub 的风险。
- **自动化测试**：核心逻辑都有测试覆盖，保证改动不会悄悄破坏已有功能。

---

## 七、明确不做的事（第一期范围之外）

为了把第一期做扎实，以下内容暂不纳入：

- **历史指标对比**：点赞、播放数每次采集会被最新值覆盖，不做时间序列追踪。
- **学校官网采集**：复旦各部门官网的内容采集是另一条独立工作流，不在本项目内。
- **网页界面 / 数据分析**：本项目只负责"采集 + 整理成 Excel"，不做可视化界面或二次数据分析。
- **多人 / 多账号切换**：一台机器对应一套登录态，不做账号共享。

## 八、技术栈一览（供参考）

纯 Python 实现，依赖均为成熟开源库：接口请求、网页自动化（登录用）、数据校验、Excel 生成、定时调度、日志等。环境用 `uv` 统一管理，`git clone` 后一条命令装好全部依赖。

代码仓库：`github.com/BlankJa/social-media-data-auto-collect`
