# ADR-0003: 不保留历史指标——覆盖写、不存时序

## Status
Accepted (2026-06-08)

## Context
点赞 / 播放 / 评论数是会随时间变化的。每条 Post 落盘成 `<post_id>.json`，再次采到同一条时这些指标可能已变。要不要把每次的指标都留下来做时序？

## Decision
**覆盖写**：同一 post_id 重采就覆盖旧 JSON，只保留最新一次快照，不存时间序列。`data/` 是「当前状态」而非「历史档案」。

## Consequences
- ＋ 存储模型简单：一条内容一个文件，增量水位线靠「文件是否存在」即可。
- ＋ 体积可控（每条 < 10 KB）。
- － 拿不到「某条视频播放量随时间的增长曲线」。如将来要做，最低成本路径是：保留覆盖写主库，另起一个按 `(post_id, 采集时间)` 追加的时序表，不动现有结构。

## Alternatives considered
- **按批次目录**（`data/<平台>/<日期>/...`）：每次采集一份全量快照。放弃——体积线性膨胀，且增量去重要跨目录扫。
- **时序追加表**：第一期用不上，过度设计。详见 spec §3.4。
