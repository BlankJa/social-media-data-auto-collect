# 设计：采集体验提升（交给非技术同事用）

**日期**：2026-06-16
**状态**：已批准，待写实施计划

## 背景与目标

四平台采集流水线第一期已交付（Task 1–19）。下一步要把它**交给做过手动 GUI 导出的非技术同事**使用。使用模式是「介于操作与拿结果之间」：日常自动跑、同事拿报表，偶尔需要同事加账号或重扫码。

针对非技术同事，确认了两个真实痛点（其余不做）：

1. **Cookie 过期无感知**——计划任务里 cookie 悄悄过期、采集静默变空，没人知道，报表慢慢变陈旧。
2. **加减账号要手编 YAML**——讲究缩进和引号，非技术同事易出错。

> 一个驱动因素：快手登录 cookie 名为 `kuaishou.server.webday7_st`，是 **7 天免登录会话**，即快手大约**每周过期一次**；而快手的过期**无法从接口响应可靠识别**（`result==2` 在「正常翻完」和「未登录」时都可能返回）。所以快手的过期必须靠 **cookie 年龄** 来预警。

**明确不做**（YAGNI）：自动推送到企业微信/邮件/桌面通知；重新扫码的一键化（`cli.py login` 现状够用）；报表交付改造（`reports/` 现状够用）。

## 方案总览

两个特性，纯 CLI 增强，**零新依赖**（复用已有的 typer / rich / openpyxl / pyyaml）：

1. **增强 `cli.py status`**——一眼看出"谁需要处理"。
2. **新增 `cli.py account` 子命令族**——免编 YAML 地增删查 + 批量导入账号。

采集端（`collect_account` 的全量/增量水位线逻辑）**不改**：新导入的账号没有 `_meta` 水位线，下次 `collect` 会自动当全量首采，之后转增量——这是现有行为，无需新代码。

## 特性 A：增强 `cli.py status`

### 行为

`cli.py status` 输出的 rich 表格新增/调整为以下列：

| 列 | 来源 |
|---|---|
| 平台 | 四平台固定 |
| Cookie 年龄 | **实时**从 `cookies/<平台>.json` 的 `saved_at` 算（不依赖上次运行）；文件不存在显示 `—` |
| 近7天 | `count_recent_posts`（已有） |
| 建议 | 见下方逻辑 |

表格下方固定提示一行：`重扫命令：cli.py login <平台>`。

### 「建议」列逻辑（优先级从高到低）

1. cookie 文件不存在 → `❌ 未登录，需扫码`
2. 上次运行 `cookie_health == "expired"`（从 `status.json` 读，仅 B站/微博/抖音可靠）→ `❌ 已过期，需重扫`
3. cookie 年龄 > 平台阈值 → `⚠️ 快过期，建议重扫`
4. 否则 → `✅ 正常`

### 年龄阈值

硬编码常量（不做配置文件）：

```python
COOKIE_AGE_WARN_DAYS = {"kuaishou": 6, "bilibili": 25, "weibo": 25, "douyin": 25}
```

快手 6 天 = 7 天会话提前 1 天预警；其余平台会话较长，25 天为保守默认。

### 实现要点

- 新增纯函数 `cookie_advice(platform, cookie_path, last_run_health) -> (age_text, advice)`，输入可控、可单测，不碰 I/O 之外的副作用。
- `status` 命令即使 `status.json` 不存在也能跑（近7天/建议仍可由 cookie 文件 + data/ 算出）；缺失的部分降级显示。
- cookie 年龄用已有的 `cookies.cookie_age()`。

## 特性 B：`cli.py account` 子命令族

所有写操作走**安全的结构化 YAML 读写**：load 现有 list → 改 → dump 回，并重新写好该平台的表头注释（说明 `account_id` 含义）。所有 add/import 都**按 `account_id` 去重**。

### `cli.py account list`

打印一张表，列出四平台所有账号（平台 / account_id / 名称）。

### `cli.py account add [platform] [account_id] [name]`

- 省略参数时**交互式**逐项提问（选平台菜单 → 输 id → 输名称）。
- 带全参时直接执行（脚本友好）。
- 写入对应 `configs/<platform>.yaml`，去重（同 id 跳过并提示）。

### `cli.py account remove`

交互式：选平台 → 从该平台现有账号列表里选一个删除 → 确认。

### `cli.py account import <文件>`

- 按扩展名自动识别 `.xlsx`（openpyxl）或 `.csv`（标准库 csv）。
- 统一三列：`platform` / `account_id` / `account_name`（首行表头）。
- 逐行校验：
  - `platform` 必须 ∈ {bilibili, weibo, douyin, kuaishou}，否则记为无效行。
  - `account_id` 非空。
- 按平台 merge 进对应 yaml，去重。
- 输出汇总：`新增 N 个 · 跳过 M 个(已存在) · 无效 K 个(原因: row<i> <原因>)`。

### `cli.py account template [out.xlsx]`

生成带表头（`platform / account_id / account_name`）的空 xlsx 模板，给同事填。默认输出到 `account_template.xlsx`。

### 格式校验（仅警告，不拦截）

加账号/导入时，按平台对 `account_id` 做轻量格式提示：

| 平台 | 期望格式 | 不符时 |
|---|---|---|
| bilibili | 纯数字（mid） | 黄色警告 |
| weibo | 纯数字（uid） | 黄色警告 |
| douyin | `MS4w` 开头（sec_uid） | 黄色警告 |
| kuaishou | `3x` 开头（主页末段） | 黄色警告 |

警告但仍写入（防"复旦配成东北大学 mid"那类低级错，又不挡特殊情况）。

### 实现要点

- 新增 `src/collector/accounts.py`：纯函数 `load_accounts_raw / add_account / remove_account / parse_import_file / validate_account_id`，与 CLI 表现层分离，可单测。
- yaml dump 时为每个平台文件重写固定表头注释（用平台 → 注释文案的常量表）。
- 表头注释丢失是可接受代价；用常量表保证写回时一致。
