# 多账号规模化：跨平台并行采集 + 失败账号可见性

> 设计文档 · 2026-06-17

## 背景与目标

第一/二期已交付四平台单账号采集流水线（增量水位线 + Excel 渲染 + status 面板 + 账号管理）。
现在要把每平台账号数从 1 扩到 **10~50 个**（全平台合计上百）。当前 `collect` 串行遍历平台、
串行遍历账号，在这个规模下两个问题暴露：

1. **墙钟时间过长**：首采（抖音单账号曾 145 分钟）和日常增量在几十账号下串行加起来很慢。
2. **失败账号不可见**：跑完只有 `accounts_failed=2` 的计数，不知道是哪两个、为什么，无法补救。

本设计只做**跨平台并行**和**失败账号可见性**两件事。

### 已确认的范围决策

- **并行粒度：只跨平台并行**。4 个平台并发，**同一平台内账号仍串行 + 节流**，不改变对单平台的请求节奏，符合「免费、低调、不接代理池」约束，封号风险最低。
- **开账号节奏：一次性全爬完**（保持当前行为）。新账号无水位线即全量首采，不做节流/分批 onboard。跨平台并行本身已让首采 ~4x 提速。
- **隔离已基本就绪**：`collect_account` 已 per-account try/except，单账号报错只记 failed、不阻塞同平台其他账号。本设计补的是**把失败账号暴露出来**。

### 非目标（YAGNI，本期不做）

平台内并发、代理池、`onboard` 分批命令、瞬时错误重试机制。均未选中，不掺入。

## 组件 1：跨平台并行 collect

### 重构

把 `cli.collect` 中「处理单个平台」的逻辑（现 cli.py 178–220 行）抽成独立函数：

```python
def collect_platform(
    name: str, cookie_root: Path, data_root: Path, *, full: bool
) -> PlatformStatus:
    """采集单个平台的所有账号（串行 + 节流），返回该平台的 PlatformStatus。

    cookie 缺失 → 返回 cookie_health="expired" 的兜底 status。
    任何未预期异常 → 捕获并返回兜底 status，保证不拖累其他平台线程。
    """
```

行为与现状一致：加载 cookie（失败标 `expired`）、`_load_accounts`、串行遍历账号 + 节流
（`collect_account` 不变）、聚合 ok/failed/new_posts/cookie_health 成 `PlatformStatus`。

### 并行

`collect()` 改为线程池并发：

```python
from concurrent.futures import ThreadPoolExecutor, as_completed

names = list(PLATFORMS.keys()) if platform == "all" else [platform]
plat_statuses: dict[str, PlatformStatus] = {}
with ThreadPoolExecutor(max_workers=4) as ex:
    futures = {
        ex.submit(collect_platform, n, COOKIE_ROOT, DATA_ROOT, full=full): n
        for n in names
    }
    for f in as_completed(futures):
        plat_statuses[futures[f]] = f.result()
```

`status.json` 的合并 + 写入在所有线程 join 之后单线程进行（现有逻辑不变）。
`collect bilibili` 单平台时池子里只有 1 个任务，行为与现在等价。

### 为什么线程安全

- 采集是 **I/O 密集**（httpx 网络等待 + `time.sleep` 节流），等待时释放 GIL，线程能拿到真并发。
- 每个平台只写自己的 `data/<platform>/` 子树 → 无跨线程文件争用。
- `status.json` 在所有线程 join 后才写一次 → 无写竞争。
- 每个 `PLATFORMS[name]` 是各自独立的平台实例，`last_response` 只被自己线程访问 → 无共享可变状态。
- `collect_platform` 整体 try/except 兜底 → 单平台线程异常返回兜底 status，不影响其余三个。
- loguru 线程安全；日志会交错，但每行带 `platform=` 标签可分辨。

### 效果

`collect all` 墙钟时间约 4x 下降（首采与日常增量都受益）；单平台采集不变。

## 组件 2：失败账号可见性

### Schema（status.py）

```python
class FailedAccount(BaseModel):
    account_id: str
    account_name: str
    error: str            # collect_account 返回的 error，截断到约 200 字

class PlatformStatus(BaseModel):
    accounts_total: int
    accounts_ok: int
    accounts_failed: int
    new_posts: int
    new_posts_7d: int
    cookie_health: CookieHealth
    failed_accounts: list[FailedAccount] = []   # 默认 []，旧 status.json 仍可解析
```

`collect_platform` 在某账号 `result.error` 非空时，追加一条 `FailedAccount`。
`failed_accounts` 随 `status.json` 持久化——**不用重跑也能查上次谁挂了、为什么**。

### 面板展示（render_status_panel）

- 主表新增「账号」列，显示 `28/30 ✓`（ok/total），保持表格紧凑。
- 表格下方，若任一平台有失败账号，红字逐条列出：

```
❌ 失败账号：
  douyin / 复旦二号 (MS4w…6i2E)：HTTPStatusError 429 Too Many Requests
  weibo  / 某账号 (1234567)：cookie expired
```

### 向后兼容

`failed_accounts` 有默认值 `[]`，旧的 `status.json`（无此字段）仍能 `read_status` 解析。

## 测试（TDD + 纯函数，不碰网络）

1. **collect_platform 隔离**：造假平台，第 2 个账号 fetch 抛异常 → 断言 `accounts_failed=1`、
   `failed_accounts` 含该账号且带 error、其余账号正常入库。
2. **并行等价**：4 个假平台并发跑，断言合并后的 `plat_statuses` 与串行结果一致、4 个平台都在。
3. **面板渲染失败列表**：构造含 `failed_accounts` 的 `RunStatus` → 断言输出含账号名与 error。
4. **向后兼容**：旧格式（无 `failed_accounts` 字段）的 status.json 能被 `read_status` 解析。
5. **CLI 冒烟**：已有 `test_cli_smoke.py` 继续守命令注册层。
