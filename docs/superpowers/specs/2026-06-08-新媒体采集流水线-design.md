# 新媒体采集流水线 · 设计文档

**日期：** 2026-06-08
**仓库：** `新媒体信源采集/`
**目标受众：** 实施这套流水线的人或 agent（也包括三个月后忘了所有细节的我自己）

---

## 1. 背景与目标

现状：B 站、微博、抖音、快手四个平台的"博主主页内容"目前靠人手在某 GUI 采集工具上点击、导出为 Excel，按日期手工命名归档（`*-5.13.xlsx`、`*-5.14.xlsx`、…），命名风格不统一，平台间靠人脑同步。仓库里残留了一份只做"Selenium 登录保存 cookie"的 B 站脚本和几份单条 URL 的 `*_source.json` 模板，没有真正的采集循环。

第一期要做的事：**用纯接口 + 增量逻辑代替手动点击，把四平台的视频/博文数据自动采下来，落成原始 JSON，再渲染成与现有命名风格一致的 Excel。**

### 1.1 关键约束

| 约束 | 含义 |
|---|---|
| 免费 | 不引付费代理池、不接商业反爬服务、不上付费云函数。 |
| 少人工干预 | 目标：自动每天跑、人每周看一次状态。除了 cookie 重扫，没有日常操作。 |
| 任何电脑随时启用 | 工具本身可移植：`git clone` + `uv sync` + 每平台扫码一次就能跑。Windows / macOS 都得支持。 |
| Cookie 不跨机同步 | 接受"换机器各扫一次"。降低复杂度，也消除"cookie 入 git"的安全风险。 |
| 不接第三方通知 | 不发邮件、不接 Bark / Server 酱。健康面板 `cli.py status` 是唯一的提示通道。 |

### 1.2 范围（in scope）

- 四平台账号清单管理（YAML）
- 每平台一份 selenium 登录子命令，cookie 落本机
- 增量采集（默认）+ 全量采集（`--full`）
- 原始 JSON 落盘 + Excel 渲染
- 调度（macOS launchd / Linux cron / Windows Task Scheduler）
- 日志 + `status.json` + `cli.py status` 健康面板

### 1.3 不在范围（out of scope）

- 历史指标时序对比（点赞/播放数会被每次跑覆盖）
- 复旦各部门官网的 `.otd` 采集（另一条独立工作流）
- Web UI 或第三方通知
- 数据分析、报表二次加工
- 多人/多 cookie 切换、cookie 共享

### 2.2 模块边界

每个平台子类对外只暴露一个 `Platform` 接口（在 `base.py` 里）：

```python
class Platform(Protocol):
    name: Literal["bilibili", "weibo", "douyin", "kuaishou"]

    def fetch_user_feed(
        self, account: Account, cookies: dict, since_post_id: str | None
    ) -> Iterator[RawPost]:
        """从最新一页开始翻，遇到 since_post_id 就停（增量），
           since_post_id=None 表示全量"""

    def parse(self, raw: RawPost, account: Account) -> Post:
        """把接口原始结构映射到统一 Post schema"""

    def excel_columns(self) -> list[ColumnSpec]:
        """该平台的 Excel 列定义"""

    def cookie_health(self, last_response: dict) -> Literal["ok", "warning", "expired"]:
        """根据最近一次接口响应判 cookie 健康度"""
```

`base.py` 还提供共用的：

- 增量循环（`for raw in fetch_user_feed(...): save_if_new(raw)`，遇到水位线自然终止）
- 随机退避（每账号 3–7 秒、每页 1–3 秒）
- `tenacity` 重试（指数退避，3 次）
- 风控码识别（412/403/-101/sec_warning 之类）

签名算法（`signing/`）单独模块、单独测，最易碎，不与平台主逻辑耦合。

## 3. 数据模型

### 3.1 Post Schema（pydantic）

```python
class Post(BaseModel):
    # 标识
    platform: Literal["bilibili", "weibo", "douyin", "kuaishou"]
    post_id: str
    url: str

    # 内容
    title: str                   # 必填：B 站标题 / 抖音、快手视频描述 / 微博博文正文
    caption: str | None          # 仅 B 站：视频详情页那段独立简介
    cover_url: str | None
    duration_sec: int | None
    media_type: Literal["video", "image", "text", "mixed"]
    published_at: datetime

    # 指标
    like_count: int | None
    comment_count: int | None
    share_count: int | None
    view_count: int | None
    collect_count: int | None

    # 账号快照（轻量，仅作筛选用）
    author_id: str
    author_name: str

    # 采集元信息
    fetched_at: datetime
    raw: dict                    # 原始接口响应片段，便于后续补字段
    extras: dict[str, str] = {}  # 平台 parse() 塞主 schema 未覆盖的字段
                                 # 例：author_bio（抖音）、video_url / image_urls（微博）
```

### 3.2 字段映射

| 字段 | B 站 | 微博 | 抖音 | 快手 |
|---|---|---|---|---|
| post_id | `bvid` | `mblogid` | `aweme_id` | `photoId` |
| url | `https://www.bilibili.com/video/{bvid}` | `https://weibo.com/{uid}/{mblogid}` | `https://www.douyin.com/video/{aweme_id}` | `https://www.kuaishou.com/short-video/{photoId}` |
| title | `title` | `text_raw` | `desc` | `caption` |
| caption | `desc` | — | — | — |
| cover_url | `pic` | `pic_infos.0.large.url` | `video.cover.url_list.0` | `coverUrl` |
| duration_sec | `duration` | — | `video.duration / 1000` | `duration / 1000` |
| like_count | `stat.like` | `attitudes_count` | `statistics.digg_count` | `likeCount` |
| comment_count | `stat.reply` | `comments_count` | `statistics.comment_count` | `commentCount` |
| share_count | `stat.share` | `reposts_count` | `statistics.share_count` | `shareCount` |
| view_count | `stat.view` | `reads_count` | `statistics.play_count` | `viewCount` |
| collect_count | `stat.favorite` | — | `statistics.collect_count` | — |

字段缺失一律 `None`（不强行填 0）。

### 3.3 `_meta.json`（水位线）

```json
{
  "account_id": "448167395",
  "account_name": "复旦大学",
  "last_run_at": "2026-06-08T08:00:00+08:00",
  "last_run_mode": "incremental",
  "newest_post_id": "BV1xxxxxxxxx",
  "total_posts": 384,
  "last_error": null
}
```

### 3.4 落盘约定

- 路径：`data/<platform>/<account_id>/<post_id>.json`
- 同 `post_id` **整文件覆盖写**（用 `*.tmp` + `os.replace` 原子替换）
- 因此点赞/播放/评论数等会随每次跑被覆盖，**不保留历史指标**（见 ADR-0003）
- 增量判定：从最新一页往前翻，遇到第一条「`post_id` 出现在 `_meta.newest_post_id` 之前 **且** 文件已存在」时停止。AND 条件，对人手删文件 / `_meta` 损坏鲁棒。
- `--full` 一路翻到接口返回空为止。

## 5. 登录与 Cookie

### 5.1 登录子命令

每平台一份 `src/collector/login/<平台>.py`，对外只导出 `login_and_save_cookies(out_path: Path) -> None`：

1. 用 Selenium 拉起**纯净 chromium**（不复用本机 Chrome profile，避免 Win/Mac 路径分歧与隐私污染）
2. 打开平台登录页
3. 终端打印：`请在浏览器里扫码登录，扫完关闭浏览器即可`
4. 监听关键 cookie 出现（B 站 `SESSDATA`、抖音 `sessionid`、微博 `SUB`、快手 `passToken`）
5. cookie 全了或浏览器关闭事件触发 → 写 `cookies/<平台>.json`：

```json
{
  "saved_at": "2026-06-08T08:00:00+08:00",
  "platform": "bilibili",
  "cookies": [{"name": "...", "value": "...", "domain": "...", "path": "..."}, ...]
}
```

现有的 `新媒体采集/bilibili/main.py` 里那段 Selenium + cookie 保存逻辑整理一遍当 B 站实现基础。**末尾那段与 ChatGPT 的私聊残留删掉**（与项目无关，且属敏感隐私）。

### 5.2 Cookie 健康度

每次接口调用结束后，平台子类的 `cookie_health(last_response)` 返回 `ok / warning / expired`：

- `expired`：HTTP 401 / `code == -101`（B 站）/ `status_code == 100002`（抖音风控）/ 微博跳登录页 / 快手 `result == 2`
- `warning`：连续 3 个账号 `new_posts == 0` 且未明确报 expired（接口降级返回空）
- `ok`：正常返回

`expired` 直接放弃该平台剩余账号、继续下一平台。下一次人跑 `cli.py status` 时面板上会标红，人按提示跑 `cli.py login <平台>` 补登。

## 6. Excel 渲染

### 6.1 命名约定（沿用现有风格）

| 平台 | 输出文件名 |
|---|---|
| 抖音 | `抖音-博主主页视频采集（不含置顶视频）-YYYY.MM.DD.xlsx` |
| B 站 | `B站UP主主页视频采集-YYYY.MM.DD.xlsx` |
| 微博 | `微博-博主主页的博文-YYYY.MM.DD.xlsx` |
| 快手 | `快手-个人账号视频采集（包含置顶视频）-YYYY.MM.DD.xlsx` |

> 历史文件里有混乱命名（`5!14`、`5:14`、`5.20-2`、`5.20全量`、`5.21（深圳&华南理工）`）。新流水线统一成 `YYYY.MM.DD`，不带后缀；`--full` 跑出的 Excel 文件名加 `-全量` 区分。

### 6.2 列定义（与现有 Excel 一致，含已知偏差）

实测下来的现有列序如下，**新流水线照搬**（详见 ADR-0004）。

**抖音**（13 列，源自 `抖音-博主主页视频采集（不含置顶视频）-6.1.xlsx`）

| # | 列名 | 取值 |
|---|---|---|
| 1 | 博主名称 | `author_name` |
| 2 | 博主简介 | `extras["author_bio"]`，平台 `fetch_user_feed` 时顺手抓的 user 字段；接口不给就空 |
| 3 | 视频标题 | `title` |
| 4 | 视频链接 | `url` |
| 5 | 视频点赞数 | `like_count` |
| 6 | 封面图url | `cover_url` |
| 7 | 是否置顶 | 恒为 `否`（接口本身就是"不含置顶"） |
| 8 | 发布时间 | `published_at` 转 `YYYY-MM-DD HH:MM:SS` |
| 9 | 视频时长 | `duration_sec` 转 `MM:SS` |
| 10 | 评论数 | `comment_count` |
| 11 | 收藏数 | `collect_count` |
| 12 | 转发数 | `share_count` |
| 13 | 页面网址 | `url`（与第 4 列同；保留以兼容下游） |

**B 站**（9 列，源自 `B站UP主主页视频采集 - 5.20-2.xlsx`）

| # | 列名 | 取值 | 备注 |
|---|---|---|---|
| 1 | 发布者 | `author_name` | |
| 2 | 页面源码1 | 空 | 旧工具痕迹，列保留兼容下游、值留空（见 ADR-0004） |
| 3 | 标题 | `title` | |
| 4 | 视频链接 | `url` | |
| 5 | 播放数 | `view_count` | |
| 6 | 发布时间 | `published_at` | |
| 7 | 时长 | `duration_sec` 转 `HH:MM:SS` | |
| 8 | 视频封面链接 | `cover_url` | |
| 9 | 视频封面链接_保存位置 | 空 | 旧工具有"下载到本地"步骤，本设计不下载 |

> **现有 B 站 Excel 不含点赞 / 评论 / 转发 / 收藏数。**新流水线的 schema 里照采，但渲染默认不输出，与现有表头保持一致；后续如要扩列在 ADR-0004 里加记录。

**微博**（10 列，源自 `微博-博主主页的博文 - 5.20.xlsx`）

| # | 列名 | 取值 |
|---|---|---|
| 1 | 博主昵称 | `author_name` |
| 2 | 页面网址 | 博主主页 URL `https://weibo.com/u/{author_id}` |
| 3 | 发布时间 | `published_at` |
| 4 | 详情链接 | `url` |
| 5 | 博文内容 | `title`（微博 title == 博文正文） |
| 6 | 视频链接 | `extras["video_url"]`，微博 `parse()` 从 `page_info.media_info.stream_url` 抽；非视频博文空 |
| 7 | 图片链接 | `extras["image_urls"]`，微博 `parse()` 用 `\|` 拼接 `pic_infos.*.large.url`；纯文本博文空 |
| 8 | 转发数 | `share_count` |
| 9 | 评论数 | `comment_count` |
| 10 | 点赞数 | `like_count` |

**快手**（8 列，源自 `快手-个人账号视频采集（包含置定视频） - 5.20.xlsx`）

| # | 列名 | 取值 |
|---|---|---|
| 1 | 快手个人账号链接 | `https://www.kuaishou.com/profile/{author_id}` |
| 2 | 视频封面图地址 | `cover_url` |
| 3 | 视频点赞数 | `like_count` |
| 4 | 快手个人账号名称 | `author_name` |
| 5 | 视频地址 | `url` |
| 6 | 视频标题 | `title` |
| 7 | 视频发布时间 | `published_at` |
| 8 | 视频详情链接 | `url`（与第 5 列同） |

> **现有快手 Excel 不含评论 / 转发数。**与 B 站同处理：采到但默认不渲染。

### 6.3 列定义的代码组织

`src/collector/rendering.py` 集中维护四个 `list[ColumnSpec]`，每个 ColumnSpec 是 `(列名, lambda post: 值)`。要调表头、加减列只改这一个文件，**不碰平台爬虫代码**。

提供三份注册脚本，二选一在常开机的本机装一次：

| 平台 | 脚本 | 后端 |
|---|---|---|

## 8. 日志与健康面板

### 8.1 日志

- `loguru`，按天滚动到 `logs/YYYY-MM-DD.log`
- 结构化字段：`platform | account | event | ...`
- 例：
  ```
  2026-06-08 08:00:01 | INFO | bilibili | 448167395 | collect_start mode=incremental
  2026-06-08 08:00:14 | INFO | bilibili | 448167395 | fetched_new=3 stopped_at=BV1xxxxxx
  2026-06-08 08:00:18 | WARNING | bilibili | 448167395 | cookie_warning code=-101
  ```

### 8.2 `status.json`

每次 `collect all` 结束覆盖写。schema：

```json
{
  "last_run_started_at": "2026-06-08T08:00:00+08:00",
  "last_run_finished_at": "2026-06-08T08:05:42+08:00",
  "last_run_mode": "incremental",
  "platforms": {
    "bilibili": {
      "accounts_total": 12,
      "accounts_ok": 12,
      "accounts_failed": 0,
      "new_posts": 7,
      "new_posts_7d": 48,
      "cookie_health": "ok"
    },
    "weibo":   { "...": "..." },
    "douyin":  { "...": "..." },
    "kuaishou":{ "...": "..." }
  }
}
```

### 8.3 终端面板

`cli.py status` 用 `rich.table.Table` 渲染：

```
Platform   Accounts   New (7d)   Last OK         Cookie
bilibili   12/12      48         2026-06-08      ✓
weibo      8/8        21         2026-06-07      ⚠ expiring
douyin     15/15      77         2026-06-08      ✓
kuaishou   10/10      33         2026-06-02      ✗ EXPIRED  ← cli.py login kuaishou
```

- `New (7d)` = 过去 7 天内新增（fetched_at 属于该窗）的 `*.json` 文件数。
- `Cookie` 三态：✓ ok / ⚠ warning / ✗ expired。expired 行尾给出补登命令。

## 9. 错误处理

| 情况 | 行为 |
|---|---|
| HTTP 5xx | `tenacity` 指数退避，3 次后放弃该账号、继续下一个 |
| HTTP 4xx（非风控） | 放弃该账号，写 `_meta.last_error` |
| 风控码（412/403/-101/sec_warning/result==2） | 标 cookie expired，放弃该平台剩余账号；继续下一平台 |
| Cookie 文件缺失 | 立即抛错并提示 `cli.py login <平台>` |
| 半截写崩 | `*.tmp` + `os.replace`，下次跑会通过文件存在性检测自动重补 |
| 接口字段变更（KeyError） | 平台 `parse()` 用 `pydantic` + `model_validate`，校验失败时记日志、保留 raw、不写 Post 文件，账号 `_meta.last_error` 记一行供人事后查 |

## 10. 测试

只覆盖三处，其它一律不测：

1. **`signing/`**：固定输入 → 固定 hex 输出。签名挂等于全平台瘫痪，必测。
2. **`storage.py`**：存/读/水位线判定/原子覆盖。
3. **`rendering.py`**：固定 Post list → 生成 xlsx → 断言行数、表头、首行内容。

**不测网络层。** `parse()` 对录制响应 `tests/fixtures/<平台>/sample_response.json` 做断言。接口字段变了 → 录制重做。

烟雾测试：`pytest && python cli.py status`。

## 11. 域语言（写进 `CONTEXT.md`）

- **账号 (account)**：一个平台上的博主主体，由 `(platform, account_id)` 唯一标识。
- **内容 / Post**：账号在该平台上发布的一条视频或博文，由 `(platform, post_id)` 唯一标识。
- **采集 (collect)**：一次 `cli.py collect <平台>` 的执行；增量或全量。
- **水位线 (`_meta.newest_post_id`)**：上次成功采集到的最新 post_id；增量循环遇到它就停。
- **Cookie 健康度**：`ok / warning / expired` 三态，由最近一次接口响应推断。
- **采集副产品 (artifact)**：`reports/*.xlsx` 是从 `data/` 渲染出来的，可随时重生成。

## 12. ADR 索引

- **ADR-0001** 平台抽象：为什么 `Platform` 基类 + 四子类，而不是四个独立脚本。
- **ADR-0002** Cookie 策略：为什么 selenium 扫码 + 本机存，而不是 refresh_token / 云同步。
- **ADR-0003** 不保留历史指标：覆盖写 trade-off，将来要加时序的最低成本路径。
- **ADR-0004** Excel 列保真：为什么沿用旧表头（含 B 站"页面源码1"等工具痕迹列），将来要重构列序的标准动作。

## 14. 已知风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 平台接口签名算法升级 | 该平台直接拉不到数据 | 签名独立模块 + 单测；出问题局部重写不连累其他平台 |
| 风控触发账号封禁 | cookie 直接死 | 节流退避 + 串行；最坏重扫一次 |
| `data/` 体积膨胀 | 几年后磁盘吃紧 | 每条 Post < 10 KB；100 账号 × 1000 条/年 = ~1 GB/年，可接受。真涨起来再加按月分目录 |
| `selenium` 在 CI 跑不动 | 登录子命令必须有人在场 | 本来就是给人扫码用的，不要求 CI 能跑 |
| 一台主用机器没开 | 当天的 cron 漏触发 | 周日 `--full` 自动补 |

## 15. 验收标准

第一期实施完成后，应满足：

1. `git clone` 一台干净机器 → `uv sync` → 四个平台各跑一次 `cli.py login <平台>` → 一次 `cli.py collect all` 成功跑完，`data/` 里出现非空 JSON，`reports/` 里出现命名正确的 xlsx。
2. 重跑一次 `cli.py collect all` 应跳过已有 Post（增量生效）。
3. `cli.py collect all --full` 应重抓所有历史（限于平台接口返回上限）。
4. 装上调度任务（macOS / Windows 都试一台），24 小时内自动触发一次。
5. `cli.py status` 输出符合 §8.3 的彩色表。
6. 手动改坏一份 cookie 文件 → 下次跑该平台标 `EXPIRED` → 重跑 `login <平台>` 后恢复 `ok`。
7. `pytest` 全绿。
